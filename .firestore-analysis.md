## Firestore Findings

- Auth model: Firebase anonymous auth, one uid per device/session.
- Collections in use:
  - `couples`
  - `coupleInvites`
  - `playerCouples`
  - `sessions`
  - `activities`
  - `journalEntries`
- Read patterns:
  - `playerCouples` read by doc id equal to current auth uid
  - `couples` read by doc id from the `playerCouples` mapping
  - `coupleInvites` read by doc id equal to invite code
  - `sessions` read by doc id
  - `activities` read by doc id
  - `journalEntries` queried by `coupleId == ...` and `orderBy(createdAt, desc)`
- Required composite index:
  - `journalEntries(coupleId ASC, createdAt DESC)`
- Security constraint:
  - Joining by invite code must not require broad reads on `couples`, so invite lookup is isolated into `coupleInvites`.
  - App startup should not rely on a collection query against `couples`, so current-player lookup is isolated into `playerCouples`.
